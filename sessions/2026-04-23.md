## Key insights

### Quality beats speed for agentic coding

Qwen3.6-27B at 20 t/s is 3.5x slower than the MoE Qwen3.5-35B-A3B at 71 t/s. But the quality benchmarks tell a different story:

| Benchmark | Qwen3.6-27B (20 t/s) | Qwen3.5-35B-A3B equiv | Qwen3.6-35B-A3B (MoE) |
|-----------|----------------------|----------------------|----------------------|
| SWE-bench Verified | **77.2** | ~75 (Qwen3.5-27B) | 73.4 |
| Terminal-Bench 2.0 | **59.3** | ~42 (Qwen3.5-27B) | 51.5 |
| SkillsBench | **48.2** | ~27 (Qwen3.5-27B) | 28.7 |
| Claw-Eval Pass^3 | **60.6** | ~46 (Qwen3.5-27B) | 50.0 |

The 27B dense model beats models with 15x its parameter count. It ties Claude 4.5 Opus on Terminal-Bench (59.3) and beats it on Claw-Eval Pass^3 (60.6 vs 59.6).

A smarter model needs fewer turns. Even at 3.5x slower token generation, solving a task in 3 turns instead of 6 can result in comparable wall-clock time — and failed attempts cost the most time of all.

### Architecture: why it's memory-efficient despite being dense

Qwen3.6-27B uses a hybrid attention architecture: 64 layers total, but only 16 use full attention. The other 48 use Gated DeltaNet linear attention with fixed-size state. This means KV cache cost is ~1/4 of a standard 27B model (~64 KB/token vs ~256 KB/token). At 32k context, total RSS is only 15 GB.

### Quantization barely matters for this model

All 4-bit variants (MLX 4bit, MXFP4, NVFP4, TurboQuant) decode within 3% of each other at 20-21.6 t/s. The model is bandwidth-limited: every token reads all 27B parameters. 8-bit is 40% slower (12 t/s) because it doubles the bandwidth demand. Ollama's Q4_K_M GGUF is 35% slower (12.2 t/s) than its NVFP4 format (20.1 t/s).

### mlx-vlm is broken for qwen3_5 architecture

Both mlx-lm and mlx-vlm claim `qwen3_5` support. mlx-lm works perfectly. mlx-vlm returns HTTP 500 on inference despite loading the model successfully. Use mlx-lm for this model.

### 20 t/s usability assessment

- Single-shot tasks (questions, explanations): usable, ~3x faster than reading speed
- Multi-turn agentic coding (10 turns): ~7 minutes generation time — painful for interactive use
- Recommendation: use for quality-critical tasks where getting it right matters more than speed. Use the MoE (71-90 t/s) for interactive/exploratory work.
